# Session: ghcache-fetch-only-watch-loop

## Goal
Stop the watch loop from clobbering HEAD and over-fetching. Originally `checkout_pr_branches=true` expanded to N `git fetch origin <branch> && git reset --hard FETCH_HEAD` invocations per repo per poll, all racing on one working directory.

## Current State
Shipped. 75 tests pass. Watch loop now:
- Emits one `CheckoutTask` per repo (not per branch).
- Clones if missing, otherwise runs a single `git fetch origin` per repo per cycle -- only when that repo had activity this cycle.
- Never touches HEAD. Activity = any row inserted by `events::sync` (PushEvent, PullRequestEvent, everything in /events).
- CLI `ghcache checkout <repo> <branch>` keeps explicit fetch + `git reset --hard origin/<branch>` as a manual override.

## Problem/Context
Four bugs fixed:
1. HEAD clobbered -- N branches, one dir, last task wins race.
2. Concurrent `tokio::spawn` git procs hit the same `.git/index.lock`.
3. PR branches never skipped -- `branch` table only holds `sync_branches`, so `sha_map` miss -> refetch every cycle.
4. N+1 INSERT INTO checkout per branch per cycle.

User rejected adding a `repo_fetch` watermark table ("what why") and pointed at the existing `dirty_prs` signal from `events::sync` -- "1 fetch per N reasons in a poll cycle". Then dropped the reset path entirely: "can we not reset? i want to just fetch origin that is fine".

## Solution/Approach
- `sync::run()` now returns `HashSet<(owner, name)>` of repos with any activity (not just PR-tagged activity).
- `events::sync` returns `EventsResult { had_activity, dirty_prs }` -- had_activity flips on any row inserted.
- `CheckoutTask` collapsed to `{ owner, name, fs_owner, is_dirty }`. No `reset_to`.
- `checkout::checkout_all(staging, tasks)` spawns one worker per repo: clone if missing else fetch-if-dirty else skip.
- Dropped `pr_head_refs()` helper + test. No schema change. No `record_reset`. No `sha_map`.
- CLI `checkout_one` kept as a separate explicit-reset path.

Docs: README `Watch` section + `Watch`/`Checkout` long_about in `main.rs` rewritten to match the new semantics.

## Key Insights
- The existing `dirty_prs` signal in events sync was the watermark all along. No new schema needed.
- Watch loop and CLI `checkout` have different semantics now: watch fetches only, CLI resets. Worth separating since the flags that drive them used to conflate the two.
- `checkout_on_sync` and `checkout_pr_branches` both now mean "fetch on activity". Kept distinct for config stability but they collapse to the same code path.
- Event-type filter for "what counts as activity" is broad (any /events row). Could tighten to PushEvent+PullRequestEvent only if over-fetching shows up, but current trigger is still correct (1 fetch per cycle regardless of N events).

## Open Questions
- Should `checkout_on_sync` and `checkout_pr_branches` merge into a single `fetch_on_activity` flag? Kept split for backward compat; user did not ask to collapse.
- `had_activity` currently fires on any event row. PushEvents/PullRequestEvents would be a tighter signal. Not worth touching until there's evidence of wasteful fetches.
