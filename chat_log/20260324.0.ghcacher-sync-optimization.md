## Problem/Context
ghcacher is a read-only local cache for GitHub data. The core design principle is cheapness -- minimize GitHub API calls. The main mechanism is:
1. Sync events (ETag-gated, usually 304, nearly free)
2. Extract PR numbers from PullRequestEvent / PullRequestReviewEvent payloads
3. Batch those dirty PR numbers into one aliased GraphQL query per repo
4. Full sweep only on startup and explicit `ghcache sync`

## Solution/Approach
- `gh` CLI is the transport (inherits auth, no octocrab)
- SQLite WAL mode, ETag/Last-Modified conditional requests throughout
- `change_log` append-only table as local IPC/webhook bus
- `ghcache serve` broadcasts change_log via Unix socket
- `ghcache-client` Rust library with typed queries + Subscriber
- `GitHubClient` trait + `MockGhClient` treats shell boundary like HTTP -- records calls, queues fixture responses
- Full vs targeted PR sync encoded in call_log endpoint label: `graphql:prs:full/owner/name` vs `graphql:prs:targeted/owner/name`
- Two separate rate limit pools tracked: REST and GraphQL cost points

## Key Insights
- No write-back to GitHub. This is a cache, read-only.
- GitHubClient trait is object-safe, all sync fns take &dyn GitHubClient
- MockGhClient lives in #[cfg(test)] block in gh.rs as pub(crate)
- Events API lossy past ~300 events -- full sweep on startup handles this
- branch.sha from branch sync is the authoritative signal for checkout updates, not polling git
- Checkout path repeats repo name: staging/myorg/main/backend, staging/myorg/staging/backend
- `ghcache sync` always full_sweep=true; watch() first_run=true then false
