## Problem/Context
Sprefa (cross-repo code intelligence daemon) needs to discover on-disk checkout paths from ghcacher to feed to git2 for tree walking across branches. The ghcache-client library had no way to query the checkout table. Additionally, the README lacked documentation on how to find the DB file and query it directly from external tools.

## Solution/Approach
- Added `Checkout` type (repo_slug, branch, local_path, sha, checked_out_at) to ghcache-client types
- Added `Client::checkouts(repo: Option<&str>)` query over the checkout table with repo filter
- Fixed pre-existing type inference error (E0282) in `Client::pr()`
- Added README sections: DB discovery (ghcache db-path, config resolution), direct SQL querying (CLI + sqlite3), schema overview with built-in views, change_log tailing
- Fixed all README code examples to use .await (were sync from before sqlx migration)

## Key Insights
- Sprefa does NOT need to mirror ghcacher's repo model. It has its own repo concepts. The only bridge is checkout paths.
- Integration model: sprefa calls ensure_repo() to subscribe, polls change_log for branch/checkout events, calls checkouts() to get local_path, feeds to git2.
- Both projects keep their own configs. ghcacher stays dynamic + configurable, sprefa has its own [[repos]].
- The monorepo merge (sprefa + ghcacher) is deferred but the sqlx migration (d2a07a9) cleared the libsqlite3-sys version conflict blocker.
- ghcache DB is WAL-mode so concurrent reads from external tools are safe while ghcache watch is running.
