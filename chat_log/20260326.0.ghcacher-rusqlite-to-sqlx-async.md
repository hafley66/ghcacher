# Session: ghcacher-rusqlite-to-sqlx-async

## Goal
Migrate the entire ghcacher project from rusqlite (sync) to sqlx (async) + tokio, so that ghcache-client and the main binary share the same db/http stack as the sprefa project (sqlx 0.8, tokio 1, reqwest 0.12).

## Current State
- ghcache-client has already been rewritten to use sqlx + tokio + reqwest (query.rs, tail.rs, cmd.rs, lib.rs, Cargo.toml all done)
- The main ghcacher binary (src/) still uses rusqlite throughout -- causes a hard libsqlite3-sys link conflict with sqlx in the same workspace
- 0 source files in src/ have been touched yet for the async migration
- All source files in src/ have been read and are understood

## Problem/Context
rusqlite 0.38 links libsqlite3-sys 0.36. sqlx 0.8 links libsqlite3-sys 0.28. Two packages linking the same native library with different major versions is a hard Cargo workspace conflict. The only fix is to put both on the same stack (sqlx).

The user wants small binaries, same stack everywhere: sqlx 0.8 + tokio 1 + reqwest 0.12.

## Solution/Approach
Full migration of all 17 src/ files:

**Key architectural decisions already made:**
- `GitHubClient` trait becomes async via `async-trait = "0.1"` crate (needed for `dyn GitHubClient`)
- `MockGhClient` in tests: `RefCell` ﻗ `Arc<Mutex<>>` (RefCell is not Send, required for async)
- DB layer: `Connection` ﻗ `SqlitePool` at entry points; sub-functions that run inside transactions take `&mut sqlx::SqliteConnection`
- Transactions: `pool.begin().await?` ﻗ `tx.commit().await?` (drop = rollback)
- Subprocess calls: `std::process::Command` ﻗ `tokio::process::Command`
- Rate limit sleep: `std::thread::sleep` ﻗ `tokio::time::sleep`
- HTTP server (cmd.rs): raw `TcpListener` + threads ﻗ `tokio::net::TcpListener` + `tokio::spawn`
- setup.rs: `inquire` (sync TUI) stays sync, wrapped in `tokio::task::spawn_blocking`
- main.rs: gets `#[tokio::main]`

**Per-file plan:**
1. `Cargo.toml` (root): rusqlite ﻗ sqlx 0.8 (runtime-tokio, sqlite), add tokio 1, add async-trait 0.1
2. `src/db.rs`: SqlitePool-based, all fns async, write fns take `&mut SqliteConnection`
3. `src/gh.rs`: async trait + impl, tokio::process::Command, tokio::time::sleep
4. `src/sync/branches.rs`: async, &mut SqliteConnection
5. `src/sync/events.rs`: async, &mut SqliteConnection
6. `src/sync/notifications.rs`: async, &mut SqliteConnection
7. `src/sync/prs.rs`: async, &mut SqliteConnection
8. `src/sync/mod.rs`: async, pool.begin() for transactions, tokio::spawn for parallelism
9. `src/query/sql.rs`: async sqlx
10. `src/query/prs.rs`: async sqlx
11. `src/query/events.rs`: async sqlx
12. `src/query/notifications.rs`: async sqlx (was already updated but still uses rusqlite internally)
13. `src/query/mod.rs`: async routing
14. `src/checkout.rs`: async, tokio::spawn, tokio::process::Command, sqlx
15. `src/cmd.rs`: tokio::net::TcpListener, tokio::spawn, sqlx for fetch_changes
16. `src/setup.rs`: spawn_blocking wrapper
17. `src/main.rs`: #[tokio::main], all .await

## Key Insights
- `async-trait` needed because `dyn GitHubClient` is used in sync functions -- native RPITIT doesn't support dyn dispatch
- `MockGhClient` uses `RefCell<VecDeque<...>>` which is !Send -- must change to `Mutex` for async context
- The per-repo transaction pattern in sync/mod.rs: `pool.begin().await?` + pass `&mut *tx` to sub-fns
- `db::log_call` and `db::set_poll_state` are called inside `gh.call()` -- they take `&mut SqliteConnection` so they work both inside and outside transactions
- ghcache-client/src/tail.rs test helper uses `include_str!("../../src/schema.sql")` -- this cross-crate path still works in workspace
- `checkout.rs` uses `stmt.raw_bind_parameter()` for dynamic IN clauses -- sqlx equivalent is `QueryBuilder::push_bind()` in a loop
- `cmd.rs` broadcast loop owns TcpStream per SSE client -- in tokio, use `OwnedWriteHalf` from `tokio::net::TcpStream::into_split()`
- `throttle_if_needed` currently does `std::thread::sleep` which would block tokio thread -- must become `tokio::time::sleep`
- `setup.rs` uses `inquire` (sync terminal TUI) -- wrap entire setup::run call in `spawn_blocking` from main
